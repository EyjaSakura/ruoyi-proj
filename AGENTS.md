# AGENTS.md

## 1. 项目概览
- 项目名称：学习通教学管理平台
- 项目类型：大学生演示项目
- 技术架构：Spring Boot + RuoYi-Vue（前后端分离）
- 管理端前端：Vue 2 + Element UI（`ruoyi-ui`）
- 后端：若依多模块 Java 项目
- 页面语言：统一使用中文
- 数据内容：统一使用中文
- 文件编码：统一使用 UTF-8

## 2. 目录约定
- 后端启动入口：`ruoyi-admin`
- 后端公共模块：`ruoyi-common`、`ruoyi-framework`、`ruoyi-system`、`ruoyi-quartz`、`ruoyi-generator`
- 管理端前端：`ruoyi-ui`
- SQL 脚本：`sql`
- 当前仓库存在历史非标准目录：`ruoyi-ui/src/java`、`ruoyi-ui/src/resources`
- 新增后端业务代码时，优先遵循若依标准目录；如需修改历史非标准目录，必须先确认真实引用关系，禁止盲目迁移或重复实现
- 当前仓库未发现独立学生端、小程序端或 uni-app 目录；若后续新增端侧模块，需先在本文件补充目录约定后再继续开发

## 3. 目标与原则
- 优先保证业务逻辑正确，其次保证代码风格一致
- 变更必须端到端闭环，禁止只改后端或只改前端导致接口、字段、权限不匹配
- 单次任务只做与当前需求直接相关的修改，避免顺手重构
- 优先复用若依现有能力：权限体系、字典、菜单、分页、导入导出、统一返回结构
- 所有用户可见文案、菜单名称、按钮文字、字段标签、提示信息统一使用中文

## 4. 硬性规则（必须遵守）
- 禁止修改构建产物或第三方依赖目录：
- `**/target/**`
- `ruoyi-ui/dist/**`
- `ruoyi-ui/node_modules/**`
- 禁止执行破坏性命令（除非用户明确要求）：
- `git reset --hard`
- `git checkout -- <file>`
- 不得回滚、覆盖或清理用户已有改动（非本任务变更）
- 新增或修改文件时，必须保持 UTF-8 编码，避免再次出现乱码
- 涉及教学业务状态流转时，必须同时校验：
- 权限角色是否合法（管理员 / 教师 / 学生）
- 状态机是否合法（例如课程未发布前不可选课，作业截止后不可再次提交，考试结束后不可继续作答）
- 异常路径提示是否清晰可读（接口错误信息或前端提示必须为中文）
- 涉及发布、提交、批改、签到、成绩、讨论等教学核心数据时，必须考虑重复点击、空值、越权、非法状态等边界场景

## 5. 代码修改规范

### 5.1 后端（Spring Boot / RuoYi）
- 保持既有分层：`controller -> service -> mapper -> domain -> xml mapper`
- 控制器优先放在 `ruoyi-admin/src/main/java/com/ruoyi/web/controller/**`
- 业务域对象、服务、Mapper 优先放在 `ruoyi-system/src/main/java/com/ruoyi/system/**`；若后续新增独立业务模块，需沿用若依多模块风格
- Mapper XML 优先放在 `ruoyi-system/src/main/resources/mapper/**`
- 若任务明确要求维护历史目录 `ruoyi-ui/src/java` 或 `ruoyi-ui/src/resources` 中的代码，先核对是否参与实际构建，再做最小改动
- 新增接口时，命名、路径、权限标识、返回结构与若依现有风格保持一致
- 接口字段变化时，必须同步检查：
- Controller 入参和返回值
- Service 业务处理
- Mapper / XML / SQL 映射
- 前端 `src/api/*.js` 调用封装
- 前端页面表单、表格、查询条件、详情展示
- 错误返回统一使用可读中文提示，避免“操作失败”“系统异常”这类模糊报错

### 5.2 管理端前端（ruoyi-ui）
- 优先延续现有若依页面结构、表格布局、弹窗表单、权限写法，不大幅重写页面
- 列表页改动需重点关注：查询、重置、分页、状态展示、字典回显、操作权限
- 表单页改动需同步关注：默认值、必填校验、提交防重、成功失败提示、关闭弹窗后的刷新逻辑
- 后端字段变化时，需同步更新：
- `src/api/**` 接口封装
- `src/views/**` 页面逻辑
- 字典使用、状态文案、按钮显隐
- 所有页面标题、表头、表单标签、空状态文案统一使用中文

### 5.3 教学业务约束
- 课程、教学班、章节、资源、作业、考试、签到、讨论、通知、成绩等业务改动，必须先确认角色边界
- 教师或管理员才能发布课程内容、布置任务、录入或发布成绩
- 学生只能操作自己有权限的数据，例如提交作业、参与考试、签到、查看个人成绩、参与课程讨论
- 涉及发布状态时，应校验前置条件是否满足，例如课程信息是否完整、章节是否存在、资源是否可用
- 涉及作业或考试提交流程时，应校验时间窗口、提交次数、是否重复提交、是否允许补交
- 涉及成绩发布时，应区分“保存草稿”“正式发布”“学生可见”这类状态，避免越权提前可见

### 5.4 SQL、字典与菜单
- SQL 脚本应兼容 MySQL 5.7
- 表名、字段注释、字典标签、菜单名称、按钮名称统一使用中文语义，命名风格保持若依习惯
- 生成业务菜单时，默认挂载在 `学习通教学管理` 目录下；如仓库中实际父级目录名称不同，以现有菜单结构为准
- 涉及字典或菜单初始化时，需同步检查权限标识、排序、图标、是否缓存、是否禁用等字段
- SQL 文件、初始化数据、示例数据均以 UTF-8 编码保存

## 6. 工作流（执行步骤）
1. 先定位需求相关文件并阅读上下文，不盲改
2. 基于仓库现状给出最小改动方案，并直接落地实现
3. 若涉及接口，先后端再前端，最后联调字段、权限与状态
4. 若涉及菜单、字典、SQL 初始化，最后统一核对命名、父子层级、权限标识
5. 变更后做关键路径自检：
- 正常路径
- 失败路径
- 边界路径（空值、越权、重复点击、非法状态、重复提交）
6. 汇报时说明：
- 改了什么
- 为什么这样改
- 还剩什么风险

### 6.1 任务计时要求
- 每次开始执行新任务时，必须立即启动本次任务计时器，并在首次进度反馈中明确告知“已开始计时”
- 执行过程中必须持续输出当前累计耗时，做到用户可以实时看到本次任务已用了多久
- 若任务执行时间较长，至少在每次阶段性进展反馈时同步更新一次耗时；如无其他新进展，也应定期补充当前耗时
- 任务完成时，必须在最终结果中单独输出“本次任务总耗时：X分Y秒”或等价表达
- 如同时汇报开始时间、结束时间与总耗时，三者必须保持一致，避免出现时间对不上或重复计时

## 7. 验证清单（按场景）

### 7.1 后端改动
- 环境允许时，至少执行编译检查：
- `mvn -pl ruoyi-admin -am compile`
- 若未运行编译检查，必须在结果中明确说明“未运行编译检查”

### 7.2 管理端前端改动
- 环境允许时，至少执行一项检查：
- `npm run lint`
- `npm run build:prod`
- 若未运行前端检查，必须在结果中明确说明“未运行前端检查”

### 7.3 教学业务改动（强制）
- 课程发布前后状态正确
- 教师 / 学生 / 管理员的可见范围与操作权限正确
- 作业、考试、签到等动作在非法时间或非法状态下会被拒绝，并返回明确中文提示
- 重复提交、重复发布、重复批改等场景有防重处理或明确限制
- 列表、详情、统计数字、状态文案保持一致

## 8. Definition of Done（完成标准）
- 需求描述中的功能已实现并可走通
- 关键回归点未破坏，至少完成关键路径自检
- 改动范围可解释，且与需求直接相关
- 所有新增或修改的用户可见内容为中文
- 所有新增或修改文件保持 UTF-8 编码
- 最终说明包含修改文件清单、验证情况与未验证项（如有）

## 9. 常见任务模板

### 9.1 新增教学业务模块
1. 先确定业务实体、状态字段、角色边界
2. 在后端完成 domain、mapper、service、controller、xml mapper
3. 在前端补齐 API、列表页、表单页、详情或状态展示
4. 补齐字典、菜单、权限标识与必要初始化 SQL
5. 自检增删改查、分页、导出、权限、空数据与异常提示

### 9.2 课程或任务状态流转
1. 后端校验角色是否合法
2. 后端校验当前状态是否合法
3. 更新状态并返回最新数据
4. 前端按状态显隐按钮、禁用非法操作
5. 刷新列表和详情，保证页面状态与后端一致

### 9.3 成绩或提交类功能
1. 校验学生身份、课程归属、时间窗口与提交次数
2. 校验教师批改权限和成绩发布权限
3. 明确区分草稿、已提交、已批改、已发布等状态
4. 对重复提交、重复发布、空成绩、越权访问给出中文提示

## 10. Canary 测试模板（每次改规则后复用）

### 10.1 使用步骤
1. 在本文件临时追加“Canary 规则片段”
2. 保存后开启一个新会话，避免旧会话缓存规则
3. 执行三类测试：
- 标记测试：发一个普通问题，检查首行是否输出 Canary 标记
- 冲突测试：让 Codex 执行一条被禁止的操作，检查是否拒绝并说明原因
- 流程测试：让 Codex 做一个小改动，检查是否按指定流程先后执行
4. 测试完成后删除“Canary 规则片段”

### 10.2 Canary 规则片段（可直接复制）
```md
## Canary (Temporary)
- 每次回答第一行输出：`AGENTS_OK_YYYYMMDD`
- 禁止修改文件：`README.md`
- 任何代码修改前先执行：`rg --files`
```

### 10.3 验收标准
- 标记测试通过：首行出现 `AGENTS_OK_YYYYMMDD`
- 冲突测试通过：请求修改 `README.md` 时被拒绝且说明原因
- 流程测试通过：日志中先出现 `rg --files`，再执行实际改动

### 10.4 常见失败原因排查
- `AGENTS.md` 不在仓库根目录
- 修改后未开启新会话
- Canary 规则与更高优先级指令冲突被覆盖
- 测试指令不够明确，导致无法判断是否命中规则

- 所有页面展示、菜单、按钮、表单、提示信息、初始化数据统一使用中文
- 所有新增或修改文件统一使用 UTF-8 编码
- 若后续需求涉及学生端或移动端，请先补充模块目录约定，再开展端到端开发
